1. ¿Qué es despliegue?

  • Despliegue: Es poner una app en internet para que la usen los usuarios.
  • Entorno: Lugar donde la app funciona (pruebas, producción).
  • Seguridad: La app debe estar protegida y lista para usarse.

¿Por qué es importante?

  • Permite que los usuarios accedan a la app.
  • Asegura que todo funcione correctamente.

2. Checklist para desplegar una app web

  • Artefacto: El archivo final de la app (JAR, WAR, Docker).
  • Configuración: Variables y secretos para cada ambiente.
  • Base de datos: Migraciones y conexión lista.
  • Infraestructura: Servidores, red y contenedores preparados.
  • Observabilidad: Logs y métricas para monitorear la app.
  • Seguridad: TLS y secretos protegidos.
  • Rollback: Plan para volver atrás si hay errores.
  • Pruebas de humo: Verificar que la app funciona tras el despliegue.

¿Qué pasa si falta algo?

  • La app puede fallar o ser insegura.
  • Los usuarios pueden perder acceso.

3. Herramientas gratuitas

  • GitHub Actions: Automatiza pruebas y despliegues gratis.
  • Jenkins: Servidor para automatizar tareas.
  • Docker: Crea contenedores para la app.
  • Kubernetes: Orquesta los contenedores.

4. Herramientas de pago

  • GitLab SaaS: Automatización avanzada y soporte.
  • AWS CodeDeploy: Despliegue en la nube.
  • Heroku: Despliegue fácil y rápido.
  • Azure DevOps: Integración con servicios de Microsoft.

5. Tecnologías requeridas

  • Maven/Gradle, JAR/WAR, Docker: Herramientas para crear y empaquetar la app. Docker permite que funcione igual en cualquier servidor.
  • Seguridad (TLS, secretos, escaneo de código): Protege los datos y revisa el código para evitar errores y ataques.
  • Observabilidad (logs, métricas, trazas): Permite ver si la app funciona bien y encontrar problemas rápido.
  • Infraestructura (Linux, Nginx, balanceadores): Son los sistemas y programas que hacen posible que la app esté disponible para los usuarios.
  • Datos (migraciones, backups, pools): Ayudan a mantener la información segura y actualizada.
  • Estrategias (blue/green, canary, rolling): Formas de actualizar la app sin afectar a todos los usuarios al mismo tiempo.

6. ¿Se aconseja desplegar back y front en el mismo servidor?

  • Juntos: Útil en proyectos pequeños.
  • Separados: Mejor para seguridad y escalabilidad.

Patrones recomendados

  • Front en CDN para velocidad.
  • Back en contenedores con escalado automático.
  • Dominios separados y CORS configurado.

7. ¿Qué es Integración Continua y Despliegue Continuo?

  • CI: Cada cambio se prueba automáticamente.
  • CD: Automatiza el paso a producción.

Pipeline de ejemplo

stages: [build, test, package, push, deploy]
build:
  script: ./gradlew clean build -x test
package:
  script: docker build -t registry/app:$CI_COMMIT_SHA .
push:
  script: docker push registry/app:$CI_COMMIT_SHA
deploy:
  script: kubectl -n prod set image deploy/app api=registry/app:$CI_COMMIT_SHA

8. Pasos para desplegar microservicios en Java

  1. Conteneriza cada servicio.
  2. Sube la imagen al registry.
  3. Crea manifiestos K8s/Helm.
  4. Configura secretos y variables.
  5. Define rutas y TLS (Ingress).
  6. Monitorea con dashboards y alertas.

¿Por qué usar microservicios?

  • Permite dividir la app en partes independientes.
  • Cada parte se actualiza y escala por separado.

9. Relación entre Kubernetes y el despliegue

  • Orquestación: Gestiona y distribuye los contenedores.
  • Desacopla: Separa la app de la infraestructura.
  • Estrategias: Rolling, blue/green y canary integradas.

Flujo típico

# Construir y publicar imagen
mvn -DskipTests package && docker build -t registry/app:1.0.0 . && docker push registry/app:1.0.0
# Aplicar manifiestos
kubectl apply -f k8s/
# Ver estado
kubectl -n prod rollout status deploy/api
kubectl -n prod get pods,svc,ingress
# Rollback
kubectl -n prod rollout undo deploy/api

10. Dinámica de despliegue en una empresa

  1. PR a main dispara CI.
  2. Merge construye imagen Docker.
  3. CD a staging con pruebas.
  4. Promoción manual a prod.
  5. Despliegue canary y monitoreo.

Ambientes y monitoreo

  • Ambientes: dev, staging, prod.
  • Monitoreo con Prometheus y Grafana.
  • Trazas con OTel.

Glosario rápido

  • Blue/Green: Tener dos versiones de la app y cambiar entre ellas para actualizar sin afectar a los usuarios.
  • Canary: Lanzar la nueva versión solo a algunos usuarios para probar que funciona bien.
  • Rollback: Volver a la versión anterior si algo sale mal en el despliegue.
  • Secrets: Guardar contraseñas y datos privados de forma segura, nunca en el código.
  • Health Check: Una prueba para saber si la app está funcionando correctamente.
  • OTel (OpenTelemetry): Herramienta para ver cómo funciona la app y encontrar errores fácilmente.
